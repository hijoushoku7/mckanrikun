# CLAUDE.md

このリポジトリで作業する Claude / エージェント向けのガイドライン。

## プロジェクト概要

ローカルネットワーク向け Minecraft サーバー管理 Web アプリ。Proxmox / Traefik の管理画面のように、ブラウザから複数の Minecraft サーバー(Docker コンテナ)を起動・停止・コンソール操作・設定編集・新規作成できる。

- 詳細な要件は [`docs/requirements.md`](docs/requirements.md) を参照(常にこれを基準にする)。
- 技術スタック: Next.js (UI) + Hono (API/BFF) + Docker (`itzg/minecraft-server`) + SQLite。
- 稼働環境: Ubuntu Server(単一ホスト、管理画面と MC サーバーは同一マシン)。

## Git / GitHub ワークフロー(必須)

1. **逐次コミット & push**: 意味のある変更単位ごとに git にコミットし、リモートへ push する。大きな塊で溜め込まない。
2. **直接 main へコミットしない**: 作業はフィーチャーブランチで行う(例: `feat/phase1-auth`, `fix/...`)。
3. **プルリクエスト経由でマージ**: 変更は必ず PR を作成し、レビューを経てから main にマージする。
4. **レビュー & マージ担当エージェント**: PR のレビューとマージは専用エージェント(下記「レビュー/マージエージェント」)が担当する。実装担当エージェントは自分の PR を自分でマージしない。

### レビュー/マージエージェント
- 役割: 送信された PR をレビューし、問題なければ承認してマージする。
- レビュー観点:
  - 要件 (`docs/requirements.md`) との整合性。
  - 正しさ(バグ)、セキュリティ、不要な複雑さ。
  - **パッケージ承認ポリシー(下記)を満たしているか** = 未承認パッケージが追加されていないか。
- 指摘がある場合は PR にコメントを残し、修正を待ってから再レビュー。
- 問題なければ approve → マージ(マージ後はブランチ削除)。

## パッケージ導入ポリシー(必須)

- npm パッケージ / Docker イメージを追加する際は、**最新の安定版を使用**し、**既知の脆弱性 (CVE / npm audit / GitHub Advisory) を事前に調査**する。
- 導入前に**パッケージをリスト化してユーザーの承認を求める**。承認前に `npm install` / `docker pull` 等を実行しない。
- 承認済みパッケージは `docs/requirements.md` の「パッケージ承認リスト」で管理する。

## 実装フェーズ

実装は `docs/requirements.md` §9 のフェーズ分割に従う。各フェーズは独立したエージェントに割り当て可能な粒度で定義されている。

- **Phase 0(パッケージ承認)は他フェーズの前提となるブロッキングタスク。**
