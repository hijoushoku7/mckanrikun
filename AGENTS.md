# AGENTS.md

このリポジトリで作業する Codex / エージェント向けのガイドライン。

このファイルは `CLAUDE.md` と `.claude/agents/pr-reviewer-merger.md` の内容を Codex でも使えるように整理した追加ドキュメントです。既存の Claude 用ファイルは残し、運用ルールの原典として併用します。

## プロジェクト概要

ローカルネットワーク向け Minecraft サーバー管理 Web アプリ。
ブラウザから複数の Minecraft サーバー Docker コンテナを起動・停止・コンソール操作・設定編集・新規作成できる管理画面を提供する。

- 要件の基準: `docs/requirements.md`
- 既存運用ルール: `CLAUDE.md`
- 技術スタック: Next.js UI + Hono API/BFF + Docker `itzg/minecraft-server` + SQLite
- 稼働環境: Ubuntu Server 単一ホスト。管理画面と Minecraft サーバーは同一マシン。

## Codex 作業ルール

- 作業前に `docs/requirements.md` と関連コードを確認する。
- 既存の設計、命名、UI トークン、コンポーネント構成を優先する。
- ユーザーの未コミット変更を消さない。無関係な差分は触らない。
- 本番環境はこの開発環境では起動しない。環境変数設定やデバッグ用コンテナ起動は不要。
- 変更後は可能な範囲で `npm run build` などの検証を行い、実行可否と結果を報告する。

## Git / GitHub ワークフロー

このリポジトリでは、変更は PR 経由で main に入れる。

1. 直接 `main` にコミットしない。
2. 作業はフィーチャーブランチで行う。例: `feat/phase1-auth`, `fix/...`
3. 意味のある変更単位でコミットし、リモートへ push する。
4. 変更は PR を作成し、レビューを経てから main にマージする。
5. 実装担当エージェントは自分の PR を自分でマージしない。

Codex でコミットや push を行う場合は、ユーザーの依頼範囲に沿って実行する。既存の未コミット変更がある場合は、作業対象との差分を確認し、無関係な変更を混ぜない。

## パッケージ導入ポリシー

npm パッケージや Docker イメージを追加する場合は、導入前にユーザー承認が必要。

- 最新の安定版を使用する。
- CVE、npm audit、GitHub Advisory などで既知脆弱性を事前確認する。
- 導入予定パッケージをリスト化し、ユーザー承認を求める。
- 承認前に `npm install`、`docker pull` などを実行しない。
- 承認済みパッケージは `docs/requirements.md` の「パッケージ承認リスト」で管理する。

未承認パッケージの追加は PR レビューで Blocking とする。

## 実装フェーズ

実装は `docs/requirements.md` のフェーズ分割を基準にする。

- Phase 0 のパッケージ承認は、他フェーズの前提となる Blocking タスク。
- スコープ外の実装を混ぜない。
- 要件に疑義がある場合は、勝手に解釈せずユーザーへ確認する。

## PR レビュー / マージ担当ロール

Claude 用の定義は `.claude/agents/pr-reviewer-merger.md` にある。Codex で同等の役割を担う場合は、以下のロールとして振る舞う。

### 役割

PR をレビューし、品質基準を満たす場合のみ承認・マージする。実装担当とは責務を分ける。

### 前提

- `docs/requirements.md` を要件の基準にする。
- `CLAUDE.md` とこの `AGENTS.md` の運用ルールに従う。
- GitHub 操作は `gh` CLI を使う。

### レビュー手順

1. 対象 PR を特定する。PR 番号がなければ `gh pr list` で確認する。
2. `gh pr diff <番号>` と必要なファイルを読んで差分を把握する。
3. レビュー観点に沿って評価する。
4. 指摘を Blocking / Non-blocking に分類する。
5. 判定に応じて review、comment、merge を実行する。

### レビュー観点

- 要件整合性: `docs/requirements.md` の該当要件を満たしているか。
- 正しさ: ロジック、エッジケース、エラーハンドリングに問題がないか。
- セキュリティ: 認証/認可、機密情報、コマンド/SQL インジェクション等の問題がないか。
- パッケージ承認: 未承認の依存や Docker イメージが追加されていないか。
- 複雑さ / 重複: 不要に複雑な実装や明らかな重複がないか。
- 検証: build、typecheck、テストなど必要な確認が行われているか。

### 判定

Blocking:

- バグ
- セキュリティ欠陥
- 未承認パッケージの追加
- 要件不整合
- マージ後に明確な障害が起きる差分

Non-blocking:

- 命名、軽微な整理、将来の改善提案など、マージを止めるほどではないもの。

### Blocking がある場合

- マージしない。
- `gh pr review <番号> --request-changes --body "..."` で総評を残す。
- 必要に応じて `gh pr comment` でファイル・行・修正方針を具体的に残す。
- 実装担当へ差し戻す。レビュー担当は原則としてコードを修正しない。
- 修正・push 後に再レビュー依頼を受けたら、最初から再評価する。

### Blocking がない場合

- `gh pr review <番号> --approve --body "..."` で承認する。
- リポジトリ方針に合わせて `gh pr merge <番号> --squash --delete-branch` でマージする。
- マージ結果を報告する。

### エスカレーション

以下の場合は自動で進めずユーザーへ確認する。

- 同一 PR で 3 往復しても Blocking が解消しない。
- レビュー判断が割れる。
- 未承認パッケージの承認が必要。
- 要件自体に疑義がある。

## 報告フォーマット

PR レビュー / マージ担当として完了報告する場合は、以下を簡潔に含める。

- PR 番号 / タイトル
- 判定: approve & merged / request-changes / escalated
- Blocking 指摘サマリ
- Non-blocking 指摘サマリ
- 次のアクション

通常の実装作業では、変更ファイル、検証結果、未実施の確認事項を簡潔に報告する。
