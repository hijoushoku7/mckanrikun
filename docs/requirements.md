# Minecraft サーバー管理 Web アプリ 要件定義書

> ローカルネットワーク向け Minecraft サーバー管理コンソール
> イメージ: Proxmox / Traefik の管理画面のような Web UI

最終更新: 2026-06-25

---

## 1. 概要

同一マシン上で複数の Minecraft (Java Edition) サーバーを Docker コンテナとして起動・管理する Web アプリケーション。Proxmox / Traefik の管理画面のように、ブラウザからサーバーの起動・停止・コンソール操作・設定編集・新規作成ができる。

新規サーバー作成時は Vanilla / Forge / NeoForge / Fabric のいずれかを選び、MC バージョンを UI で選択するだけで、各ローダーの自動ダウンロード・適切な Java の選択・EULA 同意までを画面上で完結させる。

### 1.1 前提・スコープ

- **稼働環境**: 単一の Ubuntu Server ホスト。管理画面と全 MC サーバーは同一マシン上。
- **MC サーバー実体**: `itzg/minecraft-server` Docker イメージを利用。ローダー DL・JVM・EULA 処理はイメージに委譲し、本アプリは **Docker API のオーケストレーター**として振る舞う。
- **対象ネットワーク**: LAN 内。ただし認証は必須(後述)。
- **対象ローダー**: Vanilla / Forge / NeoForge / Fabric。
- **MOD 配置**: 大量 MOD jar は FTP で `/data/mods` に直接配置。**Web 画面からの MOD アップロードはスコープ外**。

### 1.2 スコープ外(今回作らないもの)

- MOD jar の Web アップロード / 一覧 / 有効無効化(FTP で運用)
- コンソールログの長期保存・全文検索(リアルタイム表示のみ)
- マルチホスト(複数マシン)管理
- CPU コア数の制限指定(メモリ上限のみ対応)

## 2. 技術スタック

| 層 | 採用 | 備考 |
|---|---|---|
| フロントエンド | Next.js (React) | 管理 UI |
| バックエンド / API (BFF) | Hono | Docker API 連携・認証・WebSocket |
| MC サーバー実行基盤 | Docker + `itzg/minecraft-server` | ローダー自動 DL・JVM・EULA |
| コンテナ操作 | Docker Engine API | Node 用 Docker クライアント経由 |
| コンソール | RCON + WebSocket | コマンド送信は RCON、ログは docker logs ストリーム |
| MOD 配置 | FTP サーバー(別コンテナ) | `/data` ボリュームを共有マウント |

> ⚠️ **パッケージ選定ポリシー**: 具体的な npm パッケージ/Docker イメージタグは「§9 パッケージ承認リスト」に集約。**インストール前に最新安定版・既知脆弱性 (CVE / npm audit) を調査し、本書にリスト化して承認を得てから導入する。** 承認前に `npm install` 等を実行しない。

## 3. 機能要件

### FR-1 サーバー一覧 / ダッシュボード
- 登録済み MC サーバーをカード/テーブルで一覧表示。
- 各サーバーの状態(起動中 / 停止 / 起動処理中 / 異常)、MC バージョン、ローダー種別、ポート、メモリ割当を表示。
- 起動 / 停止 / 再起動 ボタン。

### FR-2 コンソール操作(起動中サーバー)
- リアルタイムログ表示(`docker logs -f` 相当のストリームを WebSocket で配信)。
- コマンド入力欄からサーバーへコマンド送信(**RCON 経由**)。
- ログは画面表示のみ。長期保存・検索は不要(直近のバッファのみ保持)。

### FR-3 新規サーバー作成ウィザード
- ローダー選択: Vanilla / Forge / NeoForge / Fabric。
- MC バージョン選択: 各公式メタ API から取得した**バージョン一覧をドロップダウン表示**(§7 参照)。
- ローダーバージョン選択(Forge/NeoForge/Fabric の対応バージョン)。
- **Java タグの自動決定**: MC バージョン → 必要 Java を対応表で自動選択(§6)。ユーザーは選ばない。
- **EULA 同意チェックボックス**(同意で `EULA=TRUE` を付与)。
- メモリ上限指定(後述 FR-5)。
- 作成 → itzg コンテナを生成し初回起動でローダーを自動 DL。

### FR-4 設定編集 (settings)
- `server.properties` を **GUI フォーム**で編集(主要項目: difficulty, gamemode, max-players, motd, pvp, online-mode, view-distance 等)。
- 保存で該当サーバーの `server.properties`(または itzg の対応環境変数)へ反映。
- 反映には再起動が必要な項目があることを UI で明示。

### FR-5 リソース制限
- 作成時/編集時に**メモリ上限を指定**(コンテナの memory limit + itzg の `MEMORY`/`MAX_MEMORY`)。
- **CPU コア数の指定は不要**。

### FR-6 認証 / アクセス制御
- ログイン認証を必須化。コンソール操作は強権限のため未認証アクセス不可。
- **初期は単一管理者運用**だが、**ユーザー追加・ロール機構を最初から実装しておく**(将来の拡張性確保)。
  - ロール例: `admin`(全操作) / `operator`(コンソール・起動停止) / `viewer`(閲覧のみ)。初期は admin 1 名のみ作成。
  - ユーザー管理画面(追加/削除/ロール変更)を用意。
- API・画面はロールに応じてアクセス制御。

### FR-7 MOD 配置(FTP)
- 各 MC サーバーの `/data` を参照する FTP サーバーを用意し、`/data/mods` へ jar を直接配置できる。
- 管理画面は FTP 接続情報(ホスト/ポート/対象パス)を**表示のみ**行う。

### FR-8 ポート管理
- MC サーバーのホストポート(ゲーム用 / RCON 用)は**作成時に手動指定**。
- **使用済みポートを DB に記録**し、一覧画面でポート使用状況を表示。
- 作成時に既存サーバーと重複するポートはバリデーションで弾く(任意で空きポート候補を提示)。

## 4. 非機能要件

| 項目 | 要件 |
|---|---|
| 対象ブラウザ | モダンブラウザ(Chrome/Firefox/Edge 最新) |
| 認証 | セッション or JWT。パスワードはハッシュ保存(bcrypt/argon2 等)。ロールベースアクセス制御 (RBAC) |
| 永続化 | SQLite(単一ファイル DB)。将来的な移行を見据え ORM 経由でアクセス |
| 通信 | LAN 内。リバースプロキシ + TLS は任意(将来対応可能な構成) |
| 同時管理 | 複数 MC サーバーの同時起動を許容(メモリ制限で事故防止) |
| 可観測性 | 各コンテナの状態をポーリング/イベント購読で反映 |
| 復旧性 | 管理画面再起動後も既存コンテナを再認識(コンテナにラベル付与で識別) |
| セキュリティ | パッケージは最新・脆弱性確認済みのみ採用(§2 ポリシー) |

## 5. アーキテクチャ

```
┌────────────── Next.js (UI) ──────────────┐
│  ダッシュボード / 作成ウィザード /         │
│  コンソール(WS) / settings フォーム / 認証 │
└───────────────┬──────────────────────────┘
                │ HTTP / WebSocket
┌───────────────▼── Hono (API / BFF) ──────┐
│  認証, サーバーCRUD, コンソール中継(RCON), │
│  バージョンメタ取得, Docker オーケストレーション│
└───────────────┬──────────────────────────┘
                │ Docker Engine API
        ┌───────▼─────────────────────────┐
        │ itzg/minecraft-server × N        │
        │  TYPE / VERSION / EULA / memory  │
        │  Java タグ自動選択 / RCON 有効化   │
        │  /data ボリューム                 │
        └───────▲─────────────────────────┘
                │ 同ボリュームを共有
        ┌───────┴─────────────────────────┐
        │ FTP サーバー(/data 群をマウント) │
        └─────────────────────────────────┘
```

- コンテナはアプリ固有ラベル(例 `mc-manager.id=<uuid>`)で識別し、再起動後も再認識する。
- 各サーバーのメタ情報(名前/ローダー/バージョン/ポート/メモリ/作成日時)は管理 DB に永続化。

## 6. MC バージョン → Java タグ 対応表

itzg はタグごとに単一 JDK を同梱(`:java8` `:java17` `:java21` 等)。本アプリが MC バージョンから自動決定する。

| MC バージョン | itzg タグ |
|---|---|
| 1.20.5 / 1.21 以降 | `java21` |
| 1.18 〜 1.20.4 | `java17` |
| 1.17.x | `java17` |
| 1.16.5 以下 | `java8` |

> Forge の古い版は Java 8 必須のケースがあるため、ローダー種別 × バージョンでも上書き判定できるようにする。対応表は設定ファイルとして外出しし、後から更新可能にする。

| ローダー | 取得元 |

| Forge | Forge Maven メタデータ |

## 8. データモデル(初期案)

永続化は **SQLite**(ORM 経由で将来の DB 移行に備える)。

```
User
  id, username, password_hash, role (admin|operator|viewer), created_at, updated_at

Server
  id (uuid), name, loader_type (VANILLA|FORGE|NEOFORGE|FABRIC),
  mc_version, loader_version, java_tag,
  memory_mb, game_port, rcon_port, rcon_password,
  container_id, eula_accepted, status_cache, created_at, updated_at

PortAllocation        // 使用済みポートの記録・表示用
  port, protocol (tcp|udp), purpose (game|rcon|ftp|other),
  server_id (nullable), note, created_at
```

> ポートは `Server` から導出もできるが、FTP 等サーバー非依存のポートや手動予約も扱えるよう独立テーブルで一元管理し、使用状況一覧の表示に用いる。

## 9. 実装フェーズ(エージェント分割タスク)

各フェーズは独立したエージェントに渡せる粒度で記述。**フェーズ 0 は他フェーズの前提**(承認ゲート)。

### Phase 0 — 基盤調査・パッケージ承認(ブロッキング)
- **目的**: 採用パッケージ/イメージの最新版・脆弱性を調査し、承認リストを確定。
- **タスク**:
  - 0-1. 採用候補パッケージ(§9 承認リスト)の最新安定版バージョンを確認。
  - 0-2. 各パッケージの既知脆弱性(npm audit / CVE / GitHub Advisory)を調査。
  - 0-3. itzg/minecraft-server の利用タグ・必要環境変数を確定。
  - 0-4. 調査結果を本書「§9 承認リスト」に反映し、**ユーザー承認を取得**。
- **完了条件**: ユーザーがパッケージリストを承認。承認前に install 系コマンドを実行しない。
- **成果物**: 確定版パッケージリスト。

### Phase 1 — プロジェクト初期化 & 認証基盤
- **依存**: Phase 0 承認済み。
- **タスク**:
  - 1-1. Next.js + Hono のモノレポ/プロジェクト雛形作成。
  - 1-2. SQLite + ORM セットアップ、DB スキーマ(User/Server/PortAllocation)とマイグレーション。
  - 1-3. 認証(ログイン/セッション or JWT、パスワードハッシュ)+ 初期 admin 作成。
  - 1-4. RBAC ミドルウェア(admin/operator/viewer)で API/画面を保護。
  - 1-5. ユーザー管理画面(追加/削除/ロール変更、admin のみ)。
- **完了条件**: admin でログインし、ユーザー追加・ロール設定ができる。
- **成果物**: 起動可能なアプリ雛形 + 認証 + RBAC。

### Phase 2 — Docker オーケストレーション層
- **依存**: Phase 1。
- **タスク**:
  - 2-1. Docker Engine API クライアント接続(ソケット)。
  - 2-2. コンテナの生成/起動/停止/再起動/削除ラッパー。
  - 2-3. アプリラベルによるコンテナ識別・再認識(再起動耐性)。
  - 2-4. コンテナ状態のポーリング/イベント購読。
- **完了条件**: API 経由でテスト用コンテナを起動・停止・一覧取得できる。
- **成果物**: Docker サービス層 + ユニット動作確認。

### Phase 3 — サーバー作成ウィザード(自動 DL)
- **依存**: Phase 2。
- **タスク**:
  - 3-1. バージョンメタ取得サービス(§7)+ キャッシュ。
  - 3-2. ローダー/MC バージョン/ローダーバージョン選択 UI。
  - 3-3. MC バージョン → Java タグ 自動選択ロジック(§6、対応表外出し)。
  - 3-4. EULA 同意チェック → `EULA=TRUE`。
  - 3-5. メモリ上限入力(FR-5)。
  - 3-6. ポート手動指定 + 重複バリデーション(PortAllocation 参照)、確保したポートを記録。
  - 3-7. 入力値を itzg 環境変数へマッピングしコンテナ生成。
- **完了条件**: ウィザードから 4 ローダーそれぞれを起動でき、ポートが記録される。
- **成果物**: 作成ウィザード一式。

### Phase 4 — コンソール(リアルタイムログ + RCON)
- **依存**: Phase 2。
- **タスク**:
  - 4-1. コンテナログを WebSocket でストリーム配信(直近バッファ保持)。
  - 4-2. RCON 接続(コンテナ作成時に RCON 有効化・パスワード設定)。
  - 4-3. コマンド入力 → RCON 送信 → 結果表示。
  - 4-4. コンソール画面 UI。
- **完了条件**: 起動中サーバーへコマンド送信でき、ログがリアルタイム表示される。
- **成果物**: コンソール機能。

### Phase 5 — 設定編集 (settings GUI)
- **依存**: Phase 3。
- **タスク**:
  - 5-1. `server.properties` 読み込み/パース。
  - 5-2. 主要項目の GUI フォーム + バリデーション。
  - 5-3. 保存 → ファイル/環境変数反映、要再起動項目の明示。
- **完了条件**: 画面から主要設定を編集・反映できる。
- **成果物**: settings 編集機能。

### Phase 6 — FTP 連携 & リソース表示
- **依存**: Phase 2。
- **タスク**:
  - 6-1. `/data` 群を参照する FTP サーバーコンテナ構成(docker compose)。
  - 6-2. 管理画面に FTP 接続情報(ホスト/ポート/対象パス)を表示。
  - 6-3. メモリ割当の編集反映。
  - 6-4. ポート使用状況一覧画面(PortAllocation を表示、FTP ポートも登録)。
- **完了条件**: FTP で `/data/mods` に jar を配置でき、画面に接続情報とポート使用状況が出る。
- **成果物**: FTP 構成 + 情報表示。

### Phase 7 — 仕上げ
- **依存**: Phase 1〜6。
- **タスク**:
  - 7-1. ダッシュボードの状態表示の最終化。
  - 7-2. エラーハンドリング/トースト/ローディング。
  - 7-3. README / セットアップ手順 / docker compose 一式。
  - 7-4. 通しの動作確認(4 ローダー作成→コンソール→設定→停止)。
- **完了条件**: 通しシナリオが成功。
- **成果物**: リリース可能な一式。

## 11. 確定事項

1. **認証**: 初期は単一管理者運用。ただし**ユーザー追加・ロール (admin/operator/viewer) 機構は最初から実装**。(FR-6)
2. **DB**: **SQLite**(ORM 経由)。(§4 / §8)
3. **ポート割当**: **手動指定** + **使用済みポートを記録・表示**。重複はバリデーションで防止。(FR-8 / §8 PortAllocation)
4. **稼働環境**: Ubuntu Server。(§1.1)
